# Product Vision

## Problem

Academic paper development is still managed with fragmented file sharing, manual version naming, unstructured feedback loops, and citation handling that is easy to break. These workflows do not scale well across co-authors, revisions, reviewer responses, and research assets.

## Core Goal

ResearchGit aims to bring software engineering discipline to academic writing by treating papers as versioned, collaborative, reviewable artifacts.

## Product Intent

The project should help research teams:

- track manuscript history with clear, inspectable revisions
- collaborate safely across multiple authors and editing rounds
- manage references and citations as structured project assets
- keep research decisions, drafts, and review feedback reproducible
- enable future AI automation on top of clean project state

## Non-Goals For The First Phase

The first phase should not try to solve everything at once.

- Not a journal submission system yet
- Not a full reference manager replacement yet
- Not a generic cloud note-taking app
- Not an AI-first writing agent before core project state is reliable

## Target Users

- individual researchers managing evolving drafts
- student-supervisor writing teams
- multi-author academic collaborations
- research groups that need traceability across manuscripts, references, and revisions

## Product Principles

- version history must be understandable to non-software users
- document collaboration must be safer than ad hoc file exchange
- citations should be first-class project objects
- automation should be layered on top of a reliable core workflow
- the system should support reproducibility, not just convenience
